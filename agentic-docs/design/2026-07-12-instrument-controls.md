# Trigger-Sampled Instrument Controls

Date: 2026-07-12
Status: accepted — seven-persona design review passed 7/7 on 2026-07-12; authorizes materializing the dependency-linked phases below and implementing them only behind their stated gates. It does not authorize npm publication, release, self-merge, arbitrary per-sample automation, or any claim that an instrument is reference-faithful.

## Motivation

1. Original instruments are ranges, not single samples. Roland's TR-808 exposes bass-drum tone/decay, snare tone/snappy, cymbal tone/decay, open-hi-hat decay, and tom/conga tuning. A fixed center voicing cannot substantiate the owner's original-hardware fidelity requirement.
2. The current public `TrackOptions` owns only gain and pan, while the DSP `start_voice` path receives only instrument, MIDI note, velocity, sample rate, and seed. There is no honest path from a documented instrument control to a newly triggered voice.
3. Smuggling panel controls through MIDI velocity would corrupt the established velocity-as-performance contract and make reference campaigns irreproducible.
4. A generic modulation system is not needed. The smallest truthful capability is a typed, normalized, trigger-sampled control set: configure a track, then copy those values into each new voice without allocation or per-sample control traffic.

## Thesis

Expose instrument-owned, normalized controls through an additive generic `Track` surface and a stable numeric WASM ABI. Controls are sampled when a note is triggered, matching the practical behavior of a programmed drum-machine pattern while keeping existing active voices immutable and the audio loop fixed-state.

Each instrument owns the physical mapping from normalized `[0, 1]` values to equations and calibrated ranges. TypeScript owns names and compile-time discoverability; Rust owns validation, defaults, and sound behavior; the worklet owns sample-accurate delivery. A control is not accepted merely because it moves a parameter: tune, held-out, artifact, performance, and blind-listening gates must show that its range tracks the reference hardware.

## Evidence base

- Roland's official TR-808 technical specifications list `BD: LEVEL, TONE, DECAY`, `SD: LEVEL, TONE, SNAPPY`, `CY: LEVEL, TONE, DECAY`, `OH: LEVEL, DECAY`, and tom/conga `LEVEL, TUNING`: https://support.roland.com/hc/en-us/articles/201963539-TR-808-Technical-Specifications
- Roland's operation manual is the primary behavioral reference for the original panel and interaction model: https://cdn.roland.com/assets/media/pdf/TR-808_OM.pdf
- Werner, Abel, and Smith derive the cymbal's decay/tone behavior from the original circuit, including weakly separated tone responses and multiple envelope/filter bands; their analysis supports equation-level mappings rather than a generic EQ macro: https://pureadmin.qub.ac.uk/ws/portalfiles/portal/125044847/tr_808_cymbal_a_physically_informed_circuit_bendable_digital.pdf
- The accepted architecture requires one single-threaded WASM engine, one worklet, sample-accurate event segmentation, fixed voice state, and no allocation on the sample path: `agentic-docs/design/2026-07-11-architecture.md`.
- Current code evidence: `packages/core/src/index.ts` exposes only gain/pan; `packages/core/worklet/instruments-processor.js` already applies time-tagged control-path events between render segments; `crates/dsp/src/lib.rs` stores fixed track buses and copies track identity into new voices. The proposed event follows an existing safe seam rather than introducing JS or allocation inside DSP.
- The current corpus declares isolated center-setting recordings from one original unit and an independent original-unit arrangement, but not full knob sweeps. Provenance/redistribution are owned by `agentic-docs/licensing.md`, limitations by `evals/reference-manifest.json`, cases by `evals/cases/drums-808.json`, and the corrected canonical digests plus sealed-baseline evidence are attached to issue #29 at https://github.com/keunwoochoi/instruments.js/issues/29#issuecomment-4952343907. Scratchpad audio is intentionally absent, so the claim is limited to those content-addressed declarations and cannot by itself prove endpoint behavior. Exact physical endpoint values remain unverified and must be measured or circuit-derived before an individual mapping can pass; the design intentionally does not invent them.

## Design

### 1. Public TypeScript ownership

Add an `InstrumentControlMap` whose keys are durable `InstrumentGroup` names and whose values are instrument-specific interfaces. The first entry is `"drums-808": Drums808Controls`; fields are added only in the same slice that ships their audible mapping. The eventual interface corresponds one-to-one to the original panel: `bassDrumTone`, `bassDrumDecay`, `snareTone`, `snareSnappy`, `cymbalTone`, `cymbalDecay`, `openHatDecay`, and `tomCongaTuning`. No exported control may be accepted and ignored.

The exact additive shape is:

```ts
export interface Drums808Controls { openHatDecay: number } // first example slice; later slices add only their implemented field
export interface InstrumentControlMap { "drums-808": Drums808Controls }
type ControlsFor<I extends InstrumentGroup> = [I] extends [keyof InstrumentControlMap] ? Partial<InstrumentControlMap[I & keyof InstrumentControlMap]> : never;
interface TrackOptions<I extends InstrumentGroup = InstrumentGroup> { gain?: number; pan?: number; controls?: ControlsFor<I> }
interface Track<I extends InstrumentGroup = InstrumentGroup> { readonly instrument: I; readonly index: number; noteOn(midiPitch: number, velocity?: number, timeSeconds?: number): void; noteOff(midiPitch: number, timeSeconds?: number): void; pedal(on: boolean, timeSeconds?: number): void; set(options: Omit<TrackOptions<I>, "controls">): void; setControls(controls: ControlsFor<I>, timeSeconds?: number): void }
export type EngineRuntimeErrorCode = "stale-wasm" | "event-overflow" | "unsupported-control" | "invalid-control" | "stale-voice" | "late-event";
export interface EngineRuntimeError extends Error { readonly code: EngineRuntimeErrorCode; readonly track?: number; readonly controlId?: number }
export function isControllableInstrument(group: InstrumentGroup | string): group is keyof InstrumentControlMap;
interface Engine { /* every existing member remains unchanged */ createTrack<I extends InstrumentGroup>(instrument: I, options?: TrackOptions<I>): Track<I>; onError(cb: (error: EngineRuntimeError) => void): () => void }
interface LooseTimelineTrackOptions { gain?: number; pan?: number }
type TimelineTrackValue = LooseTimelineTrackOptions | TrackOptions<keyof InstrumentControlMap>;
type TimelineTrackOptions = Readonly<Record<string, TimelineTrackValue>> & { readonly [I in InstrumentGroup]?: TrackOptions<I> };
```

The snippet is an additive delta, not a replacement for the existing `Engine`, `Track`, or options members; exports/type fixtures assert the complete merged declaration. It shows `openHatDecay` as the concrete first-slice example, and later slices augment `Drums808Controls` only with their implemented field. `createTrack("drums-808")` therefore returns a track whose `setControls()` accepts only fields implemented at that head; calling it on a literal uncontrollable family has an uninhabitable `never` argument. The tuple-wrapped conditional is intentionally non-distributive, so `Track<InstrumentGroup>` also has a `never` control argument until the group is narrowed. Runtime JavaScript still validates the family and throws rather than silently accepting unsupported controls. Existing `Track`, `TrackOptions`, callbacks, assignments, `createTrack()` calls, loose producer family strings, and `PlayOptions.tracks: Record<string, ...>` gain/pan objects remain source-compatible and are locked by positive and `@ts-expect-error` fixtures, including widened union, dynamic-string, vendor-family, and the positive literal `{ "drums-808": { controls: { openHatDecay: 0.25 } } }`. `isControllableInstrument` is the required capability/narrowing path after a group widens dynamically. Timeline `PlayOptions.tracks` uses the known-family mapped overlay plus a string-indexed union fallback, so known literals retain their precise controls and third-party labels retain gain/pan compatibility; because TypeScript cannot subtract literal keys from `string`, a dynamic unknown key with a control-shaped value is rejected at runtime. Asynchronous processor failures are observable through `onError`; they are never console-only and they fault subsequent control calls loudly.

Every public value is finite and normalized to `[0, 1]`; `0.5` is the calibrated default. TypeScript rejects unknown names at compile time and throws synchronously on non-finite/out-of-range runtime input. Unsupported controls never silently fall back to another parameter. The three-line musician path remains `const drums = engine.createTrack("drums-808", { controls: { openHatDecay: 0.25 } }); drums.setControls({ openHatDecay: 0.75 }); drums.noteOn(46, 100);`.

This layer owns names, typing, and user errors. It does not own physical endpoint values or DSP equations.

### 2. Worklet event and ABI ownership

Add a time-tagged `control` event carrying track index, stable numeric control id, normalized value, insertion sequence, and explicit event priority. Phase A moves all scheduled event types—not only controls—from the retained JavaScript queue into a preallocated fixed-capacity Rust/WASM event ring. Worklet message ingestion copies validated numeric fields through `ij_queue_event`; `ij_process` performs sample-offset segmentation and applies events inside Rust. Equal-frame order is `track` → `control` → `pedal/off` → `on`, then insertion sequence; a control and trigger at the same timestamp therefore use the new value in both live and offline renders.

Queue ingress returns a numeric status immediately. Event-time failures set a numeric code and fixed detail fields in a preallocated engine diagnostic slot. The main thread polls diagnostics with a periodic numeric `diagnostics` message; its handler reads the slot outside `process()` and returns only numeric data, and `packages/core` constructs the `EngineRuntimeError`, invokes `onError`, and marks the affected control path faulted. No Error, string, or failure `postMessage` is constructed inside the render callback. Stale export detection still happens at initialization/ingress and fails readiness before audio starts. Numeric event/control ids and diagnostic codes are append-only and tested on both sides; names never cross the ABI. Track controls do not consume a future voice-control namespace: Phase A also assigns each triggered voice an internal generation-safe handle and reserves an additive `ij_set_voice_control(handle, id, value)` seam. A stolen/stale handle must fail instead of retargeting another voice. MPE exposure remains deferred, but this PR may not create an ABI dead end that makes continuous pressure/bend/timbre impossible later.

The main thread validates, assigns insertion sequence/priority, and sorts batches before `postMessage`. The unavoidable AudioWorklet message delivery then copies numeric tuples into the fixed Rust ring; it retains no JavaScript event array and performs no sort, merge, slice, or compaction. Bounded out-of-order live insertion is implemented inside the preallocated Rust ring. The AudioWorklet render callback makes one `ij_process` call and copies WASM output; it performs no event traversal, allocation, FFI-per-event, string lookup, diagnostic construction, or JavaScript DSP. Per-sample synthesis and event segmentation remain entirely fixed-state Rust/WASM. Memory capacity and callback work have separate bounds: Phase A measures a worst-case full arrangement plus same-quantum event density, then commits `MAX_EVENTS_PER_QUANTUM` at the largest value whose p99 stays below 50% of the 2.67 ms deadline on target mobile. Ingress tracks per-quantum occupancy and rejects an over-dense batch atomically with `event-overflow`; it never defers events, admits deadline-breaking work, or partially schedules a batch. Ring-capacity stress and the same-quantum benchmark both gate the constant.

For live tracks, `timeSeconds` is the same absolute `AudioContext.currentTime` clock used by `noteOn`; omission or `0` means apply at the next render segment after message receipt. The documented deterministic live lead is `max(20 ms, measured device postMessage p99 + two render quanta)`, while `play()` retains its 80 ms batch lead. A late timestamp applies at the next segment, increments a `lateEvents` diagnostic, and is never backdated. `PlayOptions.tracks[group].controls` is a snapshot reapplied at that call's `t0` even when a persistent group track already exists; it precedes every same-frame trigger. Offline rendering uses the same priority comparator and relative timeline origin. The supported DSP-rate domain is exactly 44,100 and 48,000 Hz; initialization rejects every other supplied live/offline context rate loudly before scheduling, so no untested rate reaches a voice. Tests cover both supported rates, rejected-rate readiness, repeated `play()` calls, same-frame control/on events, deliberately late events, and event offsets 0/1/63/127 under direct chunk sizes 1/17/64/127/128 as well as the browser's 128-frame callback.

This layer owns ordering, stale-binary detection, and ABI compatibility. It does not own control semantics.

### 3. Rust track-state ownership

Add a compact copyable control block to each `TrackBus`, initialized to instrument defaults by `set_track`. `set_control` validates track, instrument support, id, and finite normalized range, then updates only fixed track state.

`note_on` copies the track's control block into the new voice constructor. Existing active voices retain the values sampled at their trigger; changing open-hi-hat decay does not retune a tail already ringing. This is faithful to pattern-programmed panel settings and makes renders deterministic without smoothing or shared mutable reads on the sample path. It does not define performance-expression semantics: the reserved generation-safe voice handle is the compatible future target for pressure, bend, timbre, or deliberate active-voice modulation.

The default control block must be bit- or drift-equivalent to the last accepted fixed preset before the API phase can pass. Existing code that never sets controls must render unchanged.

This layer owns defaults, validation, and trigger-time snapshots. It does not own public strings.

### 4. Instrument mapping ownership

Each instrument maps its normalized controls into physically named equation parameters inside its constructor. For `drums-808`, the synthesis owners remain explicit: bass drum is a triggered bridged-T-style resonant body; snare is two resonant sections plus transistor-noise/snappy filtering; cymbal/hats are six Schmitt rectangular oscillators through circuit-informed filter/envelope bands; tom/conga are triggered resonant sections. A mapping slice must name which equation coefficient or coupled circuit block it changes and justify that paradigm against the original service/manual/paper evidence and the browser budget. Level is intentionally excluded because track gain already owns level and duplicating it would create two authorities.

Mappings must be monotonic where the hardware control is monotonic, finite and stable at 44.1/48 kHz, and calibrated across at least low/center/high settings. Those calibration anchors do not define the stability domain: deterministic property/torture tests span 0, adjacent-to-boundary values, a dense fixed grid, and seeded random simultaneous combinations across every exposed control, every routed 808 voice/note, soft-to-hard velocity, both sample rates, retriggers, and maximum supported polyphony. Weakly coupled controls, such as cymbal tone affecting several bands, remain coupled when circuit evidence says they are; the API must not pretend they are orthogonal.

Any control that moves oscillator frequency, filter brightness/resonance, drive, or another nonlinear block requires an alias-specific plan. Band-limited oscillators remain polyBLEP or an equivalently proven method; nonlinear/circuit blocks use measured oversampling when native-rate artifact sweeps show foldback. The oracle is the same deterministic equation rendered at 4× the target rate with band-limited exciters/nonlinear oversampling, then low-pass FIR-resampled to the target. After level/onset alignment, in-band candidate-minus-oracle residual must remain at or below -60 dB relative RMS with no unintended spectral line above -50 dB relative to the target peak; committed synthetic alias injections prove the gate detects violations. Tests sweep the full control × note × hard-velocity domain at both accepted rates, also measure native-rate ultrasonic energy, and fail before listening when a bright endpoint aliases. Generic MR-STFT distance is not a substitute for this gate.

Endpoint numbers and tapers live beside the DSP equations and tests after evidence exists. Documentation links to those owners instead of duplicating values.

### 5. Evaluation ownership

The reference-case schema gains an optional normalized `controls` object validated against the family. Campaign render metadata includes the resolved controls, source/head revision, eval-code revision, case/corpus/reference/MIDI digests, random seed, sample rate, block/chunk schedule, compiler identity, shipped WASM digest, and complete retained render manifest. Every adjustable voice requires low/center/high tune cases where license-clean evidence exists, at least one held-out unit or setting, artifact gates, and both-rate structural tests. Each low/center/high setting is crossed with soft/medium/hard velocity, including the brightest coupled endpoints at hard velocity, so controls cannot flatten the established velocity-to-timbre response.

The campaign freezes its case and reference digests before the hypothesis is tested. A DSP-mapping PR may not alter its own goalposts: metric, corpus-contract, reference, or case changes land as a separately reviewed prerequisite and force a new baseline. Default incumbent renders from Tone.js, smplr, and spessasynth use identical MIDI, velocities, seeds where supported, sample rates, channels, lead/tail windows, and level policy; unsupported hardware controls are recorded as unavailable rather than guessed.

`dsp-bench` runs the full arrangement at the adversarial simultaneous endpoint combination as well as defaults. Evidence reports bytes per voice, note-on cost, cycles/µs per active voice, scaling through pool capacity, whole-arrangement cost, and the existing bounded voice-stealing behavior under overload; degradation may steal the documented quietest/releasing voice but may never allocate, glitch, corrupt a tail without the policy, or exceed the pool. Endpoint gates include tail continuity, retrigger overlap, open/closed-hat choke, abrupt-gate detection, repeated-hit variation, and extended overlapping patterns; standardized auditions cover isolated hits, velocity ladders, repeated hits, choke/release interactions, and full mixes with bass, pads, and pitched tracks at default and musically relevant extremes.

L4 listening uses a preregistered immutable stimuli manifest and randomization seed. Iteration AB/ABX starts from the repository floor of five qualified listeners but the preregistration must justify the final listener/trial count with an a priori power analysis for its declared effect/non-inferiority margin. Repeated randomized trials are analyzed at participant level or with a hierarchical model, never counted as independent listeners. Hidden identity, stable level matching, raw listener responses, setup/exclusion records, and confidence intervals are mandatory; the final original-hardware gate adds hidden references and anchors under the accepted MUSHRA schema. The exact qualification rule, exclusion rule, statistic, power assumptions, margin, and frozen stimuli digests are committed before collection, and owner blind acceptance remains mandatory. Objective metrics can reject artifacts and diagnose trajectory mismatch; only that frozen human protocol can accept original-hardware fidelity.

Until full knob-sweep recordings are license-verified, circuit-derived monotonicity and center recordings may authorize implementation experiments but cannot close the fidelity gate.

## Phased plan

### Phase A — additive typed control transport

One internal PR migrates the existing JS-retained event queue and segment loop to the fixed-capacity Rust/WASM event ring, adds the internal `control` event, append-only ABI ids/status, polled numeric diagnostics, generation-safe internal voice handles, fixed Rust track state, default-preserving constructor snapshots, and live/offline/E2E tests. It exports no public control name and changes no instrument equation. The first public TypeScript generic/name ships only with the first truthful Phase B mapping.

Before Phase A implementation, a separately reviewed prerequisite freezes the no-control case set, case/corpus/reference/eval-code digests, metric version, drift statistic, numerical threshold, shipped-WASM baseline, and artifact policy. The transport PR cannot edit them; any prerequisite change invalidates and reruns the baseline. Gates: existing no-control renders are digest-identical or within that preregistered drift threshold; invalid/unsupported inputs, queue overflow, and stale voice handles fail loudly through the numeric diagnostic → `onError` path; control ordering and late-event diagnostics pass the declared clock/priority/chunk matrix; TypeScript compatibility fixtures, Rust, Chromium live/offline, stale-WASM, both-rate, harness, and seven-persona current-head review pass. Full-arrangement plus maximum same-quantum events stays below 50% of the 2.67 ms deadline at p99 with zero dropped quanta. Web Audio composition explicitly tests `createEngine({ context: toneContext.rawContext, connect: false })`, then `engine.output.connect(userEffectsNode)` inside the caller's single context; a Tone transport test converts start/pause/resume positions to absolute `AudioContext.currentTime`. The packed package reruns zero-config Vite, Next, and Webpack development/production, SSR-safe import, exports/types, tree-shakeable ESM, and the package owner's `sideEffects: false` contract in `packages/core/README.md`. A production minimal-consumer build must emit exactly one WASM and one worklet asset; all-in gzip is the sum of the individually compressed runtime core JS, WASM, and worklet assets and remains at or below 100 KB. A package diff must show no postinstall script, dependency-tree expansion, or new attribution burden. The exact README quick-start receives a browser user-activation/ready/first-note smoke test.

Mobile acceptance records exact hardware, OS, and browser, with an iPhone 8 on iOS/Safari 16.4 as the minimum WebAssembly-SIMD support floor and a Pixel 6a-class Android device on current Chrome as the mid-tier baseline. At 44.1 and 48 kHz where the device permits, each runs five minutes of the full arrangement plus maximum admitted event density with p99 callback cost below 50% of 2.67 ms and `droppedQuanta == 0`. The manual script starts from a suspended context, requires a user gesture to unlock and sound the first note, then verifies suspend/resume, background/foreground, audio-route change, and interruption recovery without silent output or a second context. Main-thread contention records `postMessage` delivery p50/p95/p99; the documented scheduled-live lead follows the formula above.

Immediate performance uses no artificial look-ahead and has a separate physical loopback gate. A wired MIDI pad and a touchscreen automation rig each emit at least 200 timestamped gestures while the device audio output is recorded back into the measurement interface; the report subtracts calibrated rig latency and retains raw event/audio timestamps. On every supported desktop/mobile floor under idle and contended main-thread conditions, input-to-audible-onset must meet p50 ≤ 20 ms, p95 ≤ 30 ms, p99 ≤ 50 ms, worst case ≤ 70 ms, and p99-minus-p50 jitter ≤ 20 ms, with zero missing/double triggers. Phase B also runs a maximum-rate same-pitch trill/repeat corpus with interleaved note-offs, overlapping tails, and forced steals; generation assertions and audio gates forbid a note-off reaching the wrong voice, steal clicks, or unintended phrase choke.

### Phase B — original TR-808 control mappings

One or more voice-bounded PRs map bass drum, snare, cymbal/open hat, and tom/conga controls. The first mapping PR also exposes the generic TypeScript surface and only the control fields implemented at that head. Each later PR adds its public field in the same slice as its audible mapping and consumes a preregistered low/center/high campaign; it cannot edit that campaign or its metric/corpus contract.

Gates per slice: the full control × soft/medium/hard velocity matrix improves named tune diagnostics without a held-out regression; dense-grid/random full-domain stability and alias sweeps pass; endpoint renders are finite, bounded, terminating, tail-continuous, retrigger/choke-correct, non-mechanical across repeated hits, and artifact-trusted at 44.1/48 kHz; 128-frame live timing stays in the pocket; the default does not regress; adversarial simultaneous endpoints remain within the full-arrangement budget with per-voice/scaling evidence. Isolated and full-mix blind stimuli must meet the preregistered directional L4 non-inferiority/preference criterion from the powered qualified-listener analysis, not merely record a result, and the producer persona must explicitly pass its 10-second dismissal test on hard/bright alias risk, soft-to-hard timbral feel, tails/chokes, and default/extreme full-mix placement. Current-head panel review, real iOS Safari, and mid-tier Android pass. Every mapping slice packs a minimal consumer and runs the same package-owner-linked Vite, Next, Webpack, SSR, exports/types, tree-shaking, side-effects, dependency/postinstall, attribution, and payload-diff gates; it reports packed WASM, core JS, worklet, and all-in gzip and remains at or below 100 KB. The final slice additionally requires the frozen hidden-reference/anchor protocol across the complete kit, independent-unit evidence, and owner acceptance.

### Phase C — documentation and durable presets

Document the normalized controls in the package owner README and add playground knobs only after Phase B defaults and ranges are accepted. If repeated user workflows justify named presets, presets contain only normalized control values and link to the control owners; every factory preset is itself a quality-bearing sound choice.

Gates: API examples compile; keyboard/screen-reader operation works; no documentation duplicates calibration constants; bundler gates pass; component and all-in gzip sizes remain under the constitutional budget. Every factory preset gets isolated-hit and standardized full-mix auditions plus a blind producer/owner acceptance result; workflow demand alone cannot ship an amateur preset. Mobile gates are repeated here but are not deferred here—every earlier transport/mapping phase already requires them.

The GitHub tracker and dependency-linked slice issues own live status. This document records the accepted contract only.

## Deferred until demanded

- Arbitrary audio-rate modulation, modulation matrices, LFOs, envelopes, and MPE routing.
- Retuning already-active voices after a control change.
- Generic string-valued parameters or plug-in-defined control ids.
- Per-note control payloads in `NoteEvent`; configure a track before the trigger instead.
- Duplicate per-instrument level controls where track gain already owns level.
- Named 808 presets before the calibrated continuous ranges are accepted.
- Controls for instruments other than `drums-808`; the map is extensible, but demand and reference evidence must precede each addition.
- UI automation recording, undo/history, MIDI-learn, and DAW-style parameter smoothing.
